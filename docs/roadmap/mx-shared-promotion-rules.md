# MX Shared Promotion Rules

This document freezes how work moves from Plasma-local scope into shared ScreenSave scope.

## Core Promotion Rule

Promote work into shared system scope only when at least one of these is true:

- two or more products genuinely need the same stable abstraction
- the work clearly belongs to a general platform, validation, or companion contract

If neither condition is met, the work remains Plasma-local.

## Clearly Shared Work

These areas are clearly shared now:

- control-plane doctrine
- queue manifests and queue rules
- wave admission and dependency arbitration
- shared proof and environment contracts
- companion integration contracts once product surfaces are stable
- platform-law updates when a general contract truly changes

## Clearly Plasma-Local Work

These areas must remain Plasma-local for now:

- Plasma transition coverage growth
- Plasma authorable set and journey behavior
- Plasma Lab behavior
- Plasma selection intelligence
- Plasma content provenance and curation operations
- Plasma output, treatment, field, and dimensional language growth
- Plasma stable-versus-experimental product decisions short of the final shared review point

## Conditional Or Hybrid Work

These areas are conditional or hybrid by design:

- `MX00`: hybrid because it freezes both master governance and product-facing control docs
- `SY20`: conditional because it exists to decide whether a real shared contract has emerged
- `SY40`: hybrid because it should connect mature product surfaces to shared tooling and companions without inventing those product surfaces early
- `PX50`: hybrid because final stable recut decisions cross product posture, proof posture, and master release truth

## Examples

Promote to shared:

- a proof or environment contract that multiple savers now need
- a companion integration surface that is genuinely shared across products
- a platform or validation rule that is no longer Plasma-specific

Keep local:

- a richer transition matrix that only Plasma currently uses
- Plasma-specific provenance metadata
- a Plasma-only Lab control
- a Plasma-only selection heuristic

Defer or split:

- work that looks reusable but only has one real consumer today
- work that mixes one shared contract question with one Plasma feature slice

## Anti-Patterns To Avoid

- designing a suite-wide abstraction around a single unfinished Plasma need
- treating BenchLab or Suite integration as permission to move product behavior into shared scope
- promoting metadata or validator concepts before at least two products need the same stable meaning
- using `[H]` ownership to hide an unclear scope boundary
- widening stable scope before evidence supports it

## Plasma Identity Protection Rule

Plasma must not accidentally redesign the whole suite.

That means:

- Plasma may discover useful abstractions first
- Plasma may justify later shared promotion with evidence
- Plasma may not redefine platform or suite doctrine just because it is the current flagship hardening surface

Shared promotion is earned by demonstrated reuse or clear platform-law status, not by product ambition.
