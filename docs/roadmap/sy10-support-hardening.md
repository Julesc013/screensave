# SY10 Support Hardening

`SY10` freezes the first shared support-hardening contract layer for the post-PL program.

It exists because the current repo already has real shared support surfaces, but they are spread across substrate notes, release validation notes, host notes, and product-local proof.
Plasma is the immediate consumer, but the contract only promotes what is already shared or clearly belongs to shared host or support law.

## Purpose

`SY10` defines shared support vocabulary and shared support boundaries for:

- hardware proof matrices
- numeric, performance, and soak evidence
- multi-monitor host and present policy
- environment safety and recovery language

This is support-hardening contract work.
It is not product implementation work, not a new host framework, and not a Plasma feature tranche.

## Why SY10 Exists Now

Current repo reality is already post-`PL14`.
The current Plasma ship posture is `GO WITH CAVEATS`.
Stable Core remains Classic-first and lower-band-first, with:

- `default_preset=plasma_lava`
- `default_theme=plasma_lava`
- `minimum_kind=gdi`
- `preferred_kind=gl11`
- `quality_class=safe`

These shared contracts must preserve that truthful stable center of gravity.

Plasma's biggest open caveats are proof and support caveats rather than missing base architecture:

- cross-hardware proof is limited
- numeric and soak evidence is bounded
- multi-monitor behavior matters to the real host posture
- environment recovery wording exists, but it is not yet frozen as a shared contract

Later `PX10` and `PX11` work should consume one clear shared contract instead of each product note inventing slightly different support language.

## Relationship To MX00, PX10, And PX11

- `MX00` owns the control plane and makes `SY10` the shared proof and environment baseline for Wave 1.
- `SY10` freezes shared support law and vocabulary only.
- `PX10` consumes the `SY10` contracts when broadening Plasma proof and caveat coverage.
- `PX11` consumes the `SY10` proof and support vocabulary when broadening transition proof and running promotion review.

`SY10` does not unblock later waves by widening product scope.
It only makes the shared support contract explicit.

## Shared Now Versus Product-Local

Shared now:

- machine and environment fingerprint vocabulary
- proof-state bucket definitions for post-PL matrices
- qualitative performance and soak evidence classes
- current shared multi-monitor host policy
- minimum shared recovery wording and support-facing safety terms

Still product-local:

- exact Plasma content, journey, and transition coverage obligations
- exact Plasma metrics, soak thresholds, and promotion decisions
- Plasma-specific multi-monitor synchronization or hero and follower behavior
- Plasma-specific fallback presets, transition clamps, or visual safety rules

Promotion rule reminder:
move work into shared scope only when two products genuinely need it or when delaying formalization would duplicate obvious platform or host law.

## Domain Summary

### Hardware Proof Matrix Conventions

`SY10` freezes the fields and status language used to describe support evidence across machines and environments.
It does not freeze product-specific coverage requirements.

### Numeric, Performance, And Soak Conventions

`SY10` freezes how the repo names evidence classes and support envelopes.
It does not create hard shared numeric SLAs.

### Multi-Monitor Host And Present Policy

`SY10` freezes the current shared host truth:

- one saver session spanning the virtual desktop in fullscreen
- one child-window preview path
- topology-change rebinding rather than per-monitor fan-out

It does not create a generalized multi-monitor feature framework.

### Environment Safety And Recovery Policy

`SY10` freezes support-facing meanings for shared safety and recovery terms.
It does not create new runtime safety logic.

## How Later Product Work Should Consume These Contracts

Later product epics should:

1. use the `SY10` shared vocabulary first
2. add product-local obligations on top of that vocabulary
3. keep product-local claims explicit when they are not yet shared
4. update product proof notes, captures, known limits, and risk surfaces in the same pass

Products should not reinterpret shared terms privately.
If a product needs a new term or a different support meaning, that question routes back through `MX`.

## Consumer Rule For PX10 And PX11

`PX10` may rely on:

- the shared hardware matrix fields and status buckets
- the shared evidence-class vocabulary
- the frozen shared multi-monitor host truth
- the shared recovery and safety wording

`PX11` may rely on:

- the same proof and evidence vocabulary
- the same current host and recovery truth when describing transition support posture

Neither product epic may treat `SY10` as permission to broaden stable scope or centralize Plasma-specific semantics.

## Scope Boundary

`SY10` is complete when later product work can consume these support contracts without having to reinvent:

- matrix status language
- environment fingerprints
- performance and soak evidence classes
- shared host multi-monitor wording
- recovery and safety wording

`SY10` is not the place to implement new behavior.
