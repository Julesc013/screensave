# ScreenSave Doctrine v1

Status: active product doctrine.

## Thesis

ScreenSave is a curated family of autonomous native generative artworks, built
from portable deterministic semantics, delivered through thin native hosts,
extended primarily through validated data, and qualified by one shared proof
system.

Short form:

```text
Portable meaning. Native delivery. Deterministic proof. Optional automation.
```

## Product-Led Boundary

ScreenSave is product-led, not portfolio-led. Portfolio coordination vocabulary
may inform development, but it must not become more prominent than the savers,
the native artifacts, or the user-facing product family.

The canonical ScreenSave rule is:

```text
Share mechanics.
Preserve meaning.
```

The platform may share fixed-step time, deterministic random-number generation,
configuration primitives, capabilities, diagnostics, surfaces, presentation
contracts, native host lifecycle, content loading, and proof hooks.

Each saver product owns its visual thesis, state model, simulation,
configuration semantics, deterministic runtime plan, reference rendering rules,
degradation policy, and visual acceptance criteria.

## Four Planes

ScreenSave has four separate planes:

- Product plane: Core, Anthology, Manager, Workbench, SDK, Labs, and
  Preservation.
- Semantic and runtime plane: VisualIntent, product-family resolver,
  product-specific specifications, deterministic runtime plans, reference and
  optional accelerated compute, surfaces, presenters, and hosts.
- Proof plane: catalog, build profiles, artifact audit, fixed-step runner,
  RGBA reference surface, deterministic capture, image comparison, lifecycle
  and performance evidence, compatibility evidence, and proof bundles.
- Development plane: Git, worktrees, CI, ScreenSave-owned commands,
  ScreenSave proof bundles, optional AIDE WorkUnits and EvidencePackets, and
  optional worker harnesses.

These planes must not collapse into one universal engine, one universal
graphics graph, one mandatory Workbench, or one AIDE-owned runtime.

## AIDE Boundary

AIDE governs bounded development activity around ScreenSave. ScreenSave governs the product.

AIDE may coordinate work, record attempts, index evidence, and propose
promotion. AIDE must not own ScreenSave product identity, visual semantics,
build graph, artifact profiles, compatibility status, proof results, or final
artistic acceptance.

Saver binaries must not depend on AIDE, Omnigent, Codex, Workbench, Manager,
accounts, services, or the network.

## Extension Rule

Validated data packages are the default extension mechanism. Arbitrary runtime
plugins are not a baseline feature.

Text, image, and video inputs may produce VisualIntent and bounded
product-specific specifications. They must not bypass product schemas, proof,
or human promotion decisions, and they must not default to unrestricted native
code generation.

## Frozen Decisions

- Core is the primary product.
- Standalone savers remain autonomous.
- Anthology remains a saver.
- Manager and Workbench remain optional.
- Product semantics remain product-owned.
- Software/reference rendering is mandatory.
- Acceleration is optional and capability-gated.
- Compatibility is artifact-specific and evidence-backed.
- Workbench uses the same proof runner as CI.
- Cross-platform support means shared semantics plus native hosts, not one
  lowest-common-denominator shell.
